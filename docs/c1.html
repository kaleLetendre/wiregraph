Contract architecture

The seam between two compartments.

A contract is the defined communication between two compartments of your system — the agreed shape that lets them work without knowing each other's internals.

It is not just microservices. A compartment is any boundary you don't reach across directly: two services, a library and its caller, two programs sharing a table. Wherever code on one side depends on a shape produced on the other, there's a contract — whether you named it or not.

The variety

Contracts wear three coats.

The seam looks different depending on how the two compartments meet — over the wire, through an API surface, or through shared state. wiregraph treats all three the same way: a token both sides name.

over the wire

Service ↔ service

Two running services talk over HTTP or a queue. The contract is the route path or the topic name — the message shape they both agree on.

api surface

Library / SDK ↔ consumer

A package exports an API; another imports it. The contract is that exported surface — the function signatures and types the consumer binds to.

shared state

Program ↔ program

Two programs never touch — they share state: an env var, a database table, a file on disk. The contract is that key's name and shape.

The invisible bug class

Rename one side. Break the other. Silently.

Change a route, a field, or a topic on the producer — and the consumer keeps compiling, keeps passing its own tests, and fails only in production. Nothing in either repo points at the other.

  • grep
    searches text. The old name is simply gone from the producer, so there's nothing left to match against the consumer.
  • call graph
    follows calls. There was never a call across the boundary, so a call graph had no edge to break — it can't warn you.
  • contract
    wiregraph held an explicit link: producer → contract → consumer. Change the token on one side and the seam stops resolving — now it's a thing you can see and query.

How wiregraph detects it

Four detectors. One rule.

wiregraph scans each repo for distinctive tokens. When the same token surfaces in 2+ repos, it mints a Contract node and wires the referencing symbols to it.

wire

HTTP routes

A route path one repo serves and another calls.

/api/register
message

Queue / topic

A topic name a producer emits and a consumer handles.

device.heartbeat
state

Shared-state env vars

An env var or key two services both read or write.

DATABASE_URL
import

Library / SDK imports

An imported package links via a direct IMPORTS edge.

import @acme/sdk

Wire, message, and shared-state matches become a Contract node with REFERENCES / WIRE edges; a resolved import becomes a direct IMPORTS edge. Detection is heuristic and the contracts are proposed, not written into your code — you review the draft, wiregraph never edits a source file.
