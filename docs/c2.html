Deep dive · contract architecture

A contract is the defined communication between two compartments.

Not just microservices. A compartment is any unit that owns its own code and talks to another across a boundary — a service, a library, a script, a job. The contract is the shape they agree on at the seam. wiregraph makes that seam a first-class node.

THE.SEAM

The variety

One idea, three shapes.

Whatever the compartments are, the contract is always the shape at the seam. Here are the three you'll actually meet.

service ↔ service

The wire

Two running services talk over HTTP or a queue. The contract is the route or the topic — a string that must match on both ends.

route · message — an HTTP path, a queue/topic name
library ↔ consumer

The exported API

A package exposes a surface; another compartment imports it. The contract is the symbol name and signature you rely on.

import — an exported name two packages share
program ↔ program

Shared state

Two programs never touch, but both read the same env var, table, or file. The contract is that key — rename it in one and the other reads nothing.

state — an env var, table, or file key both sides use

The invisible bug class

Rename one side. Silently break the other.

The seam is a plain string in two codebases. Nothing links them at the language level — so grep and a call graph both stop at the repo edge.

repo-a/publisher.pyproducer
# someone "cleans up" the topic name
def publish_order(order):
    bus.emit(
        "orders.created"
        "orders.order_created",
        order,
    )
repo-b/worker.tsconsumer
// still listening on the old name
bus.on("orders.created", (o) => {
  fulfill(o);
});
// ← never fires again. no error.
✕
grep in repo-a finds the new name; grep in repo-b finds the old one — but nothing says they were ever the same contract. A call graph resolves calls by name within a repo; across the boundary there is no call to follow. The break ships green and surfaces as a mystery in production.

How wiregraph detects it

Four detectors. One shared token.

While indexing, wiregraph runs four detectors over each repo, extracting the tokens that could cross a boundary. When a token shows up in 2+ repos, it mints a Contract node and links both sides.

wire

HTTP routes

A route path one repo serves and another calls.

→ Contract + REFERENCES
message

Queues / topics

A topic name a producer emits and a consumer handles.

→ Contract + REFERENCES
state

Shared state

An env var or key two programs both read or write.

→ Contract + REFERENCES
import

Library / SDK

An API surface one package exposes and another imports.

→ Contract + IMPORTS
heuristic · proposed, not written

Detection is a heuristic — it reads code, not a spec. wiregraph proposes contracts into the graph so Claude can trace them; it never rewrites your source. Run /wiregraph-contracts to review the inferred wires and emit a draft AsyncAPI spec you can keep.
